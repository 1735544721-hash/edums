第 3 章 系统需求分析与设计

3.1 可行性分析

系统建设前需要从技术、经济与操作三个层面评估可行性。教务管理涉及课程、班级与教师信息等结构化数据，业务流程较为固定，适合采用成熟的 B/S 架构与关系型数据库实现。通过可行性分析可明确现有技术栈是否满足排课与查询等核心场景，判断建设成本与运维投入是否可控，并评估教师与管理员在实际工作中是否易于接受，从而降低后续实施风险。

3.1.1 技术可行性

本系统后端采用 Java 17 与 Spring Boot 3.2.8，结合 MyBatis-Plus 与 MySQL 实现数据访问，控制器覆盖管理员、教师、班级、课程、开课计划、课程安排、公告与教学事故等模块。框架生态成熟，接口以 REST 形式对外提供，配合 OpenAPI/Swagger 文档便于联调与维护，开发人员掌握主流 Java Web 技术即可完成二次开发，因此技术风险较低。

3.1.2 经济可行性

系统全部依赖开源组件，无需额外授权费用，数据库采用 MySQL，部署可在普通 Linux 服务器或校园内网环境完成。硬件投入主要为数据库与应用服务，结合 Maven 构建与自动化配置可降低部署时间；同时通过统一平台替代多表格管理，减少人工统计与重复录入成本，长期运维支出可控，具有较好的经济可行性。

3.1.3 操作可行性

系统面向教师与管理员两类角色，业务操作以登录、查询与维护为主，流程清晰且符合现有教务工作习惯。教师可通过课表查询、课程计划与个人信息维护完成日常教学安排，管理员可在同一界面完成班级、课程与公告管理。接口层提供统一的错误提示与参数校验，结合分页查询降低一次性操作负担，短期培训即可上手，操作可行性良好。

3.2 系统功能需求

3.2.1 用户主要功能

教师作为主要业务用户，系统需支持教师注册、审核登录、个人信息维护与密码修改，提供状态查询与基础资料更新通道。教师端还应覆盖开课计划与授课班级查询、班级课表与个人课表浏览、课程安排与开课计划的检索筛选、教学配置参数查看，并可及时接收通知公告，保证授课计划与教务要求一致，形成“查询—执行—反馈”的日常教学闭环。

3.2.2 管理员主要功能

教务管理员负责教务数据的全局维护，需要对教师账号进行新增、审核、修改与删除，对班级、课程等基础数据执行增删改查，并组织开课计划、维护课程安排与课表数据，确保班级与教师维度一致。管理员还应管理通知公告与教学事故记录、维护教学配置参数与首页统计卡片接口，配合分页查询与数据校验掌握整体运行状态，保障排课流程、课程资源与教学管理的统一协调。

3.3 非功能性需求分析

3.3.1 性能需求

系统在学期初与排课调整阶段会出现集中查询与批量维护，性能需求重点体现在课表、课程与教师列表的快速响应以及分页查询的稳定性。服务端需保证常用查询在合理时间内返回，课程安排提交要保持数据一致，避免同一时间段冲突导致的重复写入，并保持查询结果与数据库状态实时同步。

3.3.2 安全性需求

教务数据涉及教师账号与课程安排等敏感信息，系统需要提供账号密码校验与状态控制，管理员与教师的权限边界应清晰。对关键操作如账号删除、课程安排变更等应进行参数校验与日志记录，数据库层应采用定期备份与完整性约束，保证数据在异常情况下可恢复，降低误操作风险。

3.3.3 易用性与可维护性需求

系统应提供清晰的页面与接口结构，查询与维护操作保持一致的表单与分页交互，减少教师与管理员的学习成本。后端采用 Controller、Service、Mapper 分层设计，模块划分与数据表结构一一对应，便于定位问题与扩展功能；结合 Swagger 接口文档与统一返回结构，可以提高后续维护与联调效率，保障系统可持续演进。
